# パドック・馬体情報の正確性改善計画

## 目的

G1/G2 の live 予想で使うパドック・馬体情報を、予想記事の印や主観的な買い煽りではなく、観察可能な事実に寄せる。ROI を最終指標とする運用憲法は変更しない。今回の実装は「ソース追加・ファクト抽出前の品質管理」であり、LOOSE 条件や閾値は触らない。

## 採用する情報

- 公式・構造化情報: 出走表、取消、天候、馬場、オッズ、馬体重、騎手、調教師、過去成績。
- 観察情報: 毛艶、馬体の張り、太め/絞れ、トモ、歩様、踏み込み、発汗、入れ込み、落ち着き、返し馬など、目視で確認できる表現。
- 出所が追跡できる情報: ソース名、取得時刻、品質ゲートのバージョン、採用/却下理由を残せるもの。

## 採用しない情報

- 予想語: 本命、対抗、注目、狙い、買い、馬券、推奨、妙味、穴、鉄板、軸、相手、有力、勝ち負けなど。
- 検索結果・ニュース検索スニペット: 馬名周辺の断片だけでは文脈を保証できないため採用しない。
- ページ全文からの馬名周辺抽出: レース展望や広告、別馬の文脈が混じるため採用しない。
- 「期待できる」「侮れない」など、観察ではなく評価・予測に寄った文章。

## ソース階層

- Tier A: JRA/netkeiba の構造化データ。出走表、馬体重、騎手、調教師、馬場、天候、オッズなど。
- Tier B: netkeiba などの馬ごとに構造化されたパドック欄。観察語が含まれ、予想語が含まれない場合のみ採用。
- Tier C: G1/G2 の公開パドック専用ページ。東スポなど、馬ごとの短評ブロックとして取れる場合のみ採用。
- Tier D: 検索・記事本文・レース展望。現時点では採用しない。将来、構造化コメントとして安定取得できる URL と DOM が確認できた場合のみ Tier C へ昇格する。

## 実装方針

1. `paddock_quality.py` を追加し、パドック短評の品質ゲートを一元化する。
2. `scraper.fetch_paddock_reports` では Yahoo ニュース検索 fallback を停止する。
3. netkeiba の「全文抽出」fallback は採用しない。馬ごとの構造化コメントだけを対象にする。
4. `paddock_sources.py` は G1/G2/JpnI/JpnII に限定し、東スポの構造化パドック欄のみ採用する。
5. 検索ベースの Radio NIKKEI / 日刊スポーツは、現時点では収集対象から外す。正確性より情報量を優先しない。
6. fact validator の予想語 blacklist を拡張し、万一混入しても canonical fact 化前に落とす。

## 運用ルール

- 情報が取れない馬は「不明」として扱い、推測で補完しない。
- 低品質な情報を増やして consensus を作らない。consensus は独立した観察ソースに限定する。
- ソースを増やす場合は、まず品質ゲートのテストを追加し、採用/却下の根拠を確認してから live pipeline に接続する。
